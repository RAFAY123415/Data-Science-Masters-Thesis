Question: Why do I need to include private, public IP addresses and port when creating a QoD session?

Response:

When creating a Quality of Demand (QoD) session in Network-as-Code (NaC), you need to include private and public IP addresses, and sometimes a port, for several important reasons:

1. Device Identification:
   - Devices cannot be identified by their public IPv4 address alone due to Network Address Translation (NAT) technology used in mobile networks.
   - The combination of public and private IP addresses helps uniquely identify the device.

2. NAT Traversal:
   - Mobile networks often use NAT, which means multiple devices can share the same public IP address.
   - The private IP address helps distinguish between devices behind the same NAT.

3. Accurate Traffic Routing:
   - Providing both public and private IP addresses ensures that the QoD session is applied to the correct device and its traffic.

4. Port Specification:
   - Ports are used for better control and optimized bandwidth or latency.
   - They allow for more granular traffic management, especially when specific applications or services need prioritization.

5. Flexibility:
   - While ports are optional, including them provides more flexibility in defining QoD parameters.

6. Compliance with Network Architecture:
   - This approach aligns with how mobile networks typically operate, using both public and private addressing schemes.

Here's an example of how you might specify these parameters when creating a QoD session:

```python
my_device = client.devices.get(
    ipv4_address=DeviceIpv4Addr(
        public_address="233.252.0.2",
        private_address="192.0.2.25",
        public_port=80
    ),
    ipv6_address="2001:db8:1234:5678:9abc:def0:fedc:ba98",
    phone_number="+36721601234567"
)

my_session = my_device.create_qod_session(
    service_ipv4="233.252.0.2",
    service_ipv6="2001:db8:1234:5678:9abc:def0:fedc:ba98",
    profile="QOS_L",
    duration=3600
)
```

It's important to note that while you need to provide this information, you don't always need to manually determine these values. In many cases, the SDK can help retrieve this information for identified devices.

By including these details, NaC can ensure that the QoD session is correctly applied to the specific device and its network traffic, even in complex network environments with multiple layers of addressing.

Question: How does network as code handle end-user consent?

Response:

Network as Code (NaC) handles end-user consent through two main flows: Client Initiated Backchannel Authentication (CIBA) and Authorization Code flow. Here's an overview of how consent management works in NaC:

1. CIBA Flow:
   - This is the simpler and more common flow.
   - It's mostly transparent to the application developer.
   - How it works:
     a. The application calls NaC APIs with its application keys.
     b. NaC runtime handles polling for access tokens on behalf of the application.
     c. An Authorization Server guards access to the end-user device.
     d. The Authorization Server triggers an appropriate end-user consent request.
     e. The user can approve or deny the request.
     f. Upon approval, the Authorization Server provides a client token authorized for the desired action.

2. Authorization Code Flow:
   - Used when the end-user needs to take positive action to indicate consent.
   - How it works:
     a. The application constructs an authorization URL pointing to the NaC authorization server.
     b. This URL includes the application's own redirect URI.
     c. The end-user is directed to this URL.
     d. NaC redirects the user to their operator-specific consent portal.
     e. If consent is given, the application is redirected back to its redirect URI with an operator code.
     f. This code is exchanged for a single-use access token.
     g. The access token is used with subsequent API requests to validate user consent.

3. Consent Management:
   - NaC adopts a privacy-by-design approach.
   - It complies with data protection regulations like GDPR.
   - Device owners authorize which applications or APIs can access their device.
   - Consent can be granted or revoked for specific APIs and their parameters.

4. Scope of Consent:
   - Consent is typically defined by API names (e.g., "Location Verification", "Quality of Service on Demand").
   - It may include specific resource names and allowed actions (read/write).

5. Developer Responsibilities:
   - Developers need to ensure their applications request appropriate consent.
   - They should handle cases where consent is not granted.
   - Applications must respect user privacy and data protection regulations.

6. Timeouts and Handling:
   - Since consent requests may take time, developers may need to adjust API request timeouts.
   - NaC SDKs use extended timeouts by default to accommodate this.

7. Business-to-Business (B2B) Scenarios:
   - For enterprise use cases, there's a process for bulk consent management.
   - Enterprises can provide a form listing device IDs, API scopes, authorized parties, and operators.
   - NaC administrators validate this data with operators and enable access accordingly.

Here's a simplified example of how a developer might handle consent in their code:

```python
try:
    location = device.location.retrieve()
    print(f"Device location: Latitude {location.latitude}, Longitude {location.longitude}")
except nac.ConsentRequiredException:
    print("User consent is required for this operation.")
    # Implement consent request flow here
except nac.ConsentDeniedException:
    print("User has denied consent for this operation.")
```

By implementing these consent flows, NaC ensures that user privacy is protected and that applications only access data and perform actions that users have explicitly authorized.

Question: What is a code example that uses QoD API?

Response:

Certainly! Here's a code example that demonstrates how to use the Quality of Demand (QoD) API in Network as Code (NaC). This example shows how to create a QoD session, retrieve session information, and delete a session using Python:

```python
import network_as_code as nac
from network_as_code.models.device import DeviceIpv4Addr

# Initialize the Network as Code client
client = nac.NetworkAsCodeClient(token="<your-application-key-here>")

# Create a device object
my_device = client.devices.get(
    ipv4_address=DeviceIpv4Addr(
        public_address="233.252.0.2",
        private_address="192.0.2.25",
        public_port=80
    ),
    ipv6_address="2001:db8:1234:5678:9abc:def0:fedc:ba98",
    phone_number="+36721601234567"
)

# Create a QoD session
try:
    qod_session = my_device.create_qod_session(
        service_ipv4="233.252.0.2",
        service_ipv6="2001:db8:1234:5678:9abc:def0:fedc:ba98",
        profile="QOS_L",
        duration=3600,  # Duration in seconds (1 hour)
        notification_url="https://your-server.com/notifications",
        notification_auth_token="your-auth-token"
    )
    print(f"QoD session created with ID: {qod_session.id}")
    print(f"QoS Profile: {qod_session.profile}")
    print(f"Status: {qod_session.status}")
    print(f"Expires at: {qod_session.expires_at}")

    # Get all QoD sessions for the device
    all_sessions = my_device.sessions()
    print(f"Total QoD sessions for device: {len(all_sessions)}")

    # Get a specific session by ID
    session_id = qod_session.id
    retrieved_session = client.sessions.get(session_id)
    print(f"Retrieved session status: {retrieved_session.status}")

    # Extend the session duration
    extended_session = client.sessions.extend(session_id, duration=1800)  # Extend by 30 minutes
    print(f"Session extended. New expiry: {extended_session.expires_at}")

    # Delete the QoD session
    client.sessions.delete(session_id)
    print("QoD session deleted")

except nac.exceptions.NetworkAsCodeException as e:
    print(f"An error occurred: {str(e)}")
```

8. Deleting a QoD session

   Here's an example of what a response might look like:

   ```json
   {
     "id": "congestion-sub-123",
     "device": {
       "externalId": "device123@example.com"
     },
     "congestionLevel": "MEDIUM_CONGESTION",
     "timestamp": "2023-12-01T12:00:00Z"
   }
   ```

4. Interpreting the Congestion Level:
   - NO_CONGESTION: The network is operating normally with no significant congestion.
   - MEDIUM_CONGESTION: There is some congestion in the network, which might affect performance.
   - HIGH_CONGESTION: The network is experiencing significant congestion, likely impacting performance considerably.

5. Using the Information:
   - For application logic: You can use this information to adjust your application's behavior. For example, if there's high congestion, you might delay large data transfers or reduce video quality.
   - For user experience: You can inform users about potential performance issues due to network conditions.
   - For analytics: Track congestion levels over time to understand network performance patterns.

6. Subscription-based Updates:
   The Network Insights API typically works on a subscription model. You subscribe to congestion updates for a specific device or area, and you'll receive notifications when the congestion level changes.

Question: What do I need to do to my application in order to be able to receive notifications from Network as Code?

To receive notifications from Network as Code (NaC), you need to set up a notification handling system in your application. Here's what you need to do:

1. Create a Notification URL:
   Set up an HTTP endpoint in your application that can receive POST requests. This will be your notification URL (also called a webhook or callback URL).

2. Configure Authentication:
   Implement authentication for your notification endpoint. NaC will send an auth token with each notification.

3. Subscribe to Notifications:
   Use the NaC SDK to subscribe to the specific types of notifications you want to receive.

4. Handle Incoming Notifications:
   Implement logic to process the notifications when they are received at your endpoint.

Here's a more detailed breakdown of these steps:

1. Creating a Notification URL:
   - Set up an HTTPS server with a POST endpoint, e.g., "https://example.com/notifications"
   - Ensure this endpoint is publicly accessible

2. Implementing Authentication:
   - Choose an auth token for your endpoint
   - Configure your endpoint to check for this token in incoming requests

3. Subscribing to Notifications:
   Use the NaC SDK to subscribe. Here's an example in Python:

   ```python
   import network_as_code as nac

   client = nac.NetworkAsCodeClient(token="<your-application-key-here>")

   my_device = client.devices.get("device@testcsp.net")

   my_subscription = client.connectivity.subscribe(
       event_type="org.camaraproject.device-status.v0.connectivity-data",
       device=my_device,
       max_num_of_reports=5,
       notification_url="https://example.com/notifications",
       notification_auth_token="replace-with-your-auth-token"
   )
   ```

4. Handling Notifications:
   Implement a handler for the notifications. Here's a basic example using FastAPI:

   ```python
   from fastapi import FastAPI, Header
   from pydantic import BaseModel
   from typing import Union, Optional

   app = FastAPI()

   class Notification(BaseModel):
       id: str
       source: str
       type: str
       specversion: str
       datacontenttype: str
       time: str
       data: dict

   @app.post("/notifications")
   def receive_notification(
       notification: Notification,
       authorization: Union[str, None] = Header(default=None)
   ):
       if authorization == "Bearer my-token":
           # Process the notification
           print(notification)
           return {"status": "success"}
       else:
           return {"status": "unauthorized"}
   ```

Key points to remember:

- Always use HTTPS for your notification URL to ensure security.
- The notification_auth_token you set when subscribing will be sent as a Bearer token in the Authorization header of each notification.

Question: What is the difference between network APIs and regular APIs?

Response:

Network APIs and regular APIs have some key differences, particularly in their scope, functionality, and the level of access they provide. Here's a breakdown of the main distinctions:

1. Scope and Access:
   - Network APIs: Provide direct access to mobile network capabilities and resources. They allow interaction with core network functions that are typically not accessible to regular applications.
   - Regular APIs: Generally offer access to application-level services or data, without direct interaction with the underlying network infrastructure.

2. Provider:
   - Network APIs: Typically provided by telecom operators or specialized platforms like Nokia's Network as Code (NaC).
   - Regular APIs: Can be provided by any software service, application, or platform.

3. Functionality:
   - Network APIs: Offer capabilities like Quality of Service control, network slicing, device location, network insights, and other telecom-specific functions.
   - Regular APIs: Provide a wide range of functionalities depending on the service, but generally don't include low-level network control or insights.

4. Use Cases:
   - Network APIs: Often used for IoT management, real-time applications requiring guaranteed performance, location-based services, and applications needing deep network integration.
   - Regular APIs: Used for a broad spectrum of applications, from social media integration to payment processing, without specific network optimizations.

5. Performance Impact:
   - Network APIs: Can directly influence network behavior and performance for specific applications or devices.
   - Regular APIs: Generally don't have direct impact on network performance or behavior.

6. Security and Access Control:
   - Network APIs: Often require stricter security measures and access controls due to their potential impact on network operations.
   - Regular APIs: While also requiring security, they typically don't have the same level of potential network-wide impact.

7. Standardization:
   - Network APIs: Often follow telecom industry standards (e.g., 3GPP specifications) to ensure interoperability across different networks.
   - Regular APIs: May follow various standards depending on the industry, but generally have more flexibility in their design.

8. Latency and Real-time Capabilities:
   - Network APIs: Can offer lower latency and more real-time capabilities due to direct network integration.
   - Regular APIs: May have higher latency as they often operate at the application layer.

9. Cost Model:
   - Network APIs: Often have usage-based pricing models tied to network resource consumption.
   - Regular APIs: Can have various pricing models, including free tiers, subscription-based, or pay-per-use.

10. Developer Expertise:
    - Network APIs: May require some understanding of telecom concepts and network operations.
    - Regular APIs: Generally require knowledge specific to the application domain they serve.

In summary, network APIs provide a deeper level of integration with mobile network infrastructure, offering capabilities that are not typically available through regular APIs. They enable developers to create applications that can dynamically interact with and optimize network resources, leading to enhanced performance, reliability, and innovative services in areas where network capabilities are crucial.
