Question: What if there are everyone boosting QoD?

Response:

Nokia Network-as-Code (NaC) addresses the scenario of multiple users requesting Quality on Demand (QoD) boosts by utilizing Quality of Service (QoS) profiles and session management. Networks are shared resources, and various applications and devices compete for bandwidth. To prevent poorer quality, congestion, or unfairness, the system does not allow every user to choose desired speeds with no limitations.

Instead, QoS profiles are used to manage and prioritize the flow of data. When a QoD session is requested, its status can be `REQUESTED`, `AVAILABLE`, or `UNAVAILABLE`. If the network cannot allocate the requested resources due to high demand or other limitations, the session may not become `AVAILABLE`. This ensures that network performance remains stable and managed, even when multiple boost requests are made.

Question: Why do I need to include private, public IP addresses and port when creating a QoD session?

Response:

When creating a Quality on Demand (QoD) session with Nokia Network-as-Code (NaC), including private and public IP addresses along with a public port is essential for accurate identification and precise control over the network resources.

Here's why these parameters are needed:

1.  **Device Identification (Public IP, Private IP, and Public Port):** Mobile devices often operate behind Network Address Translation (NAT). In such environments, multiple devices might share a single public IP address. To uniquely identify a specific mobile network device, you must provide its public IPv4 address, its private IPv4 address, and the public port it uses. This combination allows the network to correctly map the device's internal address to its external representation, ensuring the QoD session is applied to the intended device.
2.  **Application Server Identification (Service IP addresses):** The `service_ipv4` and `service_ipv6` (or `applicationServer` in cURL) parameters are used to identify the specific application server to which the device needs a prioritized connection.
3.  **Port Specificity (Optional `devicePorts` and `applicationServerPorts`):** While not always mandatory for a basic QoD session, specifying `devicePorts` and `applicationServerPorts` allows for granular control. By defining a list or range of ports for both the device and the application server, you can ensure that the Quality of Service enhancement is applied only to traffic associated with those particular applications or services. This optimizes bandwidth and latency for specific use cases.

Question: How does network as code handle end-user consent?

Response:

Nokia Network-as-Code (NaC) incorporates a privacy-by-design approach to handle end-user consent, ensuring compliance with data protection regulations like GDPR. Consent management is crucial for granting or revoking an application's access to APIs and their parameters, allowing authorized devices to be used safely.

Here's how NaC handles end-user consent:

**1. Core Principles and Terminology:**
*   **Legitimate Interest:** Applications must have a "legitimate interest" and a legal basis to process personal or private enterprise information.
*   **Device Owner Control:** The device owner determines which applications or APIs can access their device.
*   **Identity Management:** NaC manages various device IDs, allowing different authorization scopes (API names) for multiple device IDs and organizations.
*   **Scope:** Defines the feature or operation that is authorized, typically by the API name (e.g., `location-verification`, `qod-sessions`). It can be extended with specific resources or actions like "read" or "write."
*   **Device ID:** An identifier for the device or subscriber, such as a phone number (`36721601234567`) or an email-like identifier (`device@testcsp.net`).
*   **Authorized Party:** The private enterprise, organization, or system integrator authorized to use the devices within defined scopes.
*   **Operator:** The owner of the network accessible through NaC APIs.
*   **Purpose:** Specifies why authorization is being requested (e.g., `dpv:FraudPreventionAndDetection` for fraud prevention). This can be associated with a scope (e.g., `dpv:FraudPreventionAndDetection number-verification:verify`).

**2. Consent Flows:**

*   **Client Initiated Backchannel Authentication (CIBA) Flow (for simpler cases):**
    *   NaC handles end-user requests by using CIBA, where an Authorization Server guards access to the end-user device.
    *   NaC polls this server until a response is provided.
    *   The Authorization Server triggers an end-user consent request, which the user can approve or deny.
    *   Upon approval, a client token is provided, authorized to perform the desired action.
    *   From the developer's perspective, this flow is largely transparent; developers call APIs normally, and NaC handles the polling for access tokens. Extended API request timeouts may be necessary.

*   **Authorization Code Flow (for positive user action):**
    *   This flow is used when the end-user needs to take explicit action to indicate consent.
    *   The application developer constructs an authorization URL pointing to the NaC authorization server, including their own redirect handler.
    *   The end-user application, running on the end-user device, is directed to this URL.
    *   NaC then redirects the user to their operator-specific consent portal to approve the request.
    *   If consent is given, the application is redirected back to its redirect URI with an operator code.
    *   This code is then exchanged for a single-use access token, which is passed with subsequent API requests to validate user consent. This is particularly used for APIs like Number Verification.

**3. Business-to-Business (B2B) Scenarios:**
*   For enterprises owning multiple devices (e.g., drones in agriculture), an **Organization Admin** submits an authorization request.
*   This involves filling out a "Nokia Network as Code Consent & Authorization template" (an Excel spreadsheet).
*   The template specifies:
    *   Legal Name of the Authorizing Organization and Authorized Party.
    *   Date of Consent.
    *   Device IDs (phone numbers or external identifiers).
    *   API Scopes (e.g., `qod-sessions`, `location-retrieval`).
    *   Authorized Party (organization name as listed in the NaC Portal Dashboard).
    *   Operator name.
    *   Revocation Date.
*   The Organization Admin sends this request via email to `support.networkascode@nokia.com`.
*   NaC Admin validates the data with the operators, enabling different organizations to access their allowed devices and scopes.
*   Submitting a new request completely cleans up previous authorizations; devices not included in the new list will lose their authorization.

**4. Business-to-Consumer (B2C) Scenarios (e.g., Number Verification):**
*   The Number Verification API uses a 3-legged OAuth 2.0 flow.
*   A mobile app initiates verification by requesting and receiving an authorization URL.
*   This URL must be accessed via a mobile connection and involves several redirects through a Network Provider interface.
*   The user provides consent through this interface.
*   The SDKs simplify this by automatically issuing intermediate requests to construct valid authorization requests and create the authorization URL.

**5. Application Registration and Consent Request Screenshots:**
*   During application onboarding, developers must submit information about their business and the network capabilities they intend to use.
*   Applications are expected to ask for end-user consent for personal information (e.g., location data, phone number).
*   To validate compliance with data privacy legislation, developers must provide screenshots of each step in their application's consent request flow. These screenshots are submitted to network operators by NaC support.

Question: What do I need to do to my application in order to be able to receive notifications from Network as Code?

Response:

To enable your application to receive notifications from Nokia Network-as-Code (NaC), you need to configure a notification URL, also known as a webhook, and set up an HTTP server to listen for incoming POST requests.

Here's what your application needs to do:

1.  **Set up an HTTP Server with a POST Endpoint:**
    Your application must host an HTTP server that exposes a POST endpoint. This endpoint will be the `notification_url` you provide to NaC when subscribing to events. NaC will send event updates to this URL.

    *   **Example (Python using FastAPI):**
        ```python
        from fastapi import FastAPI, Header
        from pydantic import BaseModel
        from typing_extensions import Annotated
        from typing import Union

        app = FastAPI()

        class Notification(BaseModel):
            id: str
            source: str
            type: str
            specversion: str
            datacontenttype: str
            time: str
            # Data structure will vary based on the notification type
            # For example, for slice notifications:
            # resource: str
            # action: str
            # state: str
            # For device status notifications:
            # data: { device: Device, subscriptionId: str, terminationReason: str }

        @app.post("/notifications")
        def receive_notification(
            notification: Notification,
            authorization: Annotated[Union[str, None], Header]
        ):
            # Validate the authorization token
            if authorization == "Bearer my-token": # Replace "my-token" with your actual notification_auth_token
                # Process the incoming notification
                print(notification)
                return {"status": "success"}
            else:
                return {"status": "unauthorized"}, 401
        ```

    *   **Example (TypeScript using Express):**
        ```typescript
        import express, { Express, Request, Response } from "express";

        const app: Express = express();
        const port = process.env.PORT || 3000;

        app.use(express.json());

        // Define an interface for the expected notification structure
        // This will vary based on the notification type (e.g., SliceNotification, DeviceStatusNotification)
        interface GenericNotification {
            id: string;
            source: string;
            type: string;
            specversion: string;
            datacontenttype: string;
            time: string;
            // ... other fields specific to the notification type
        }

        app.post("/notifications", (req: Request, res: Response) => {
            const notification: GenericNotification = req.body;
            const authorizationHeader = req.headers.authorization;

            // Validate the authorization token
            if (authorizationHeader === "Bearer my-token") { // Replace "my-token" with your actual notificationAuthToken
                // Process the incoming notification object
                console.log(notification);
                res.status(200).end();
            } else {
                res.status(401).send("Unauthorized");
            }
        });

        app.listen(port, () => {
            console.log(`[server]: Server is running at http://localhost:${port}`);
        });
        ```

2.  **Use HTTPS for the Notification URL:**
    For security, it is highly recommended to use HTTPS for your `notification_url`. This ensures that the notifications sent from NaC to your application are encrypted in transit.

3.  **Provide a `notification_auth_token`:**
    When you subscribe to notifications (e.g., for a QoD session, slice, or device status), you will provide a `notification_auth_token`. This token acts as a password that NaC will include in the `Authorization` header of its POST requests to your `notification_url`. Your application's server should validate this token to ensure that the notification originated from an authorized source.

4.  **Parse the Notification Payload:**
    The incoming notification will be a JSON payload. Your application needs to parse this JSON to extract the relevant information about the event. The structure of the payload will depend on the type of notification (e.g., device status, QoD session update, slice state change).

**Where Notification URLs are Used:**
You can configure notification URLs for various NaC functionalities, including:

Question: What is a code example that uses QoD API?

Response:

The Nokia Network-as-Code (NaC) Quality on Demand (QoD) API allows you to create sessions to prioritize network traffic for specific devices and application servers. Below are code examples in Python, TypeScript, and cURL for creating a QoD session.

These examples demonstrate how to:
*   Create a Network as Code client using your application key.
*   Identify a mobile network device using its IP addresses and port.
*   Create a QoD session specifying a QoS profile, service IP, and duration.

**Python Example:**

```python
import network_as_code as nac
from network_as_code.models.device import DeviceIpv4Addr

# Create a Network as Code client using your application key
client = nac.NetworkAsCodeClient(token="YOUR_APPLICATION_KEY")

# Identify the mobile network device
device = client.devices.get(
    ipv4_address=DeviceIpv4Addr(
        public_address="1.1.1.2",
        private_address="1.1.1.2",
        public_port=80
    ),
    phone_number="+36721601234567" # The phone number does not accept spaces or parentheses
)

# Create a QoD session
# Parameters:
# service_ipv4: The IPv4 address of the application server.
# profile: The QoS profile (e.g., "QOS_L") that indicates the connection type to be prioritized.
# duration: The length of the QoD session in seconds (mandatory).
qod_session = device.create_qod_session(
```

Question: How do I interpret the network insights API response?

Response:

When interpreting the Network Insights API response from Nokia Network-as-Code (NaC), particularly for congestion levels, you will typically receive information about the network's current or predicted state for a given device or area. The primary goal of Network Insights is to provide analytics on various network aspects, helping applications react proactively to real-time network performance data.

Here's how to interpret the key elements of a Network Insights API response, focusing on congestion levels:

**1. Response Structure:**
The API can return either an array of congestion level objects (when polling historical data or multiple instances) or a single congestion object (for current status). For notifications, the congestion data is typically nested within a `data` object.

**2. Key Fields and Their Interpretation:**

*   **`level`**: This is a crucial string value indicating the congestion experienced by the device. Its possible values are:
    *   **"None"**: The network is not experiencing congestion.
    *   **"Low"**: There is minimal congestion, likely not impacting performance significantly.
    *   **"Medium"**: Moderate congestion is present, which might lead to some degradation in network performance (e.g., increased latency, reduced bandwidth).
    *   **"High"**: Severe congestion is occurring, indicating a high likelihood of significant performance degradation.

*   **`confidence`**: This integer value, ranging from 0 to 100, represents the level of confidence in the congestion level prediction. A higher number indicates greater certainty in the reported `level`. This is particularly relevant when the API provides predicted congestion data.

*   **`start` and `end`**: When retrieving historical or predicted congestion data, these fields (ISO 8601 formatted date strings or date-time objects) define the timestamp range for which the congestion data is valid or was observed.

*   **`data` (for notifications)**: In the context of congestion notifications, the `data` field is an object that encapsulates the congestion level information, including the `level` itself.

*   **Other Notification-Specific Fields (for webhooks):**
    *   **`id`**: A string value representing the event subscription identifier.
    *   **`source`**: A string identifying the source of the event, including the NaC API implementation and subscription ID.
    *   **`type`**: A string indicating the status returned by the API.
    *   **`specversion`**: A string representing the specification version.
    *   **`datacontenttype`**: Indicates the media type for the event payload encoding, typically "application/json".
    *   **`time`**: A string value (ISO 8601 format) indicating when the event occurred.

**3. Example Interpretation:**

If you receive a response like this (e.g., from `myDevice.getCongestion()` or a notification):

```json
{
  "level": "High",
  "confidence": 95,
  "start": "2023-10-27T10:00:00Z",
  "end": "2023-10-27T10:15:00Z"
}
```

You would interpret it as:
*   The network is currently experiencing **High** congestion.
*   There is a **95% confidence** in this prediction.
*   This congestion level was observed or is predicted for the period between 10:00 AM and 10:15 AM UTC on October 27, 2023.

**4. Accessing Values in Code:**

In Python, after polling for congestion levels, you might access the level like this:

```python
# Assuming 'myDevice' is a device object and 'congestion' is the response
congestion_data = await myDevice.getCongestion()
# If it returns an array, access the first object
first_congestion_object = congestion_data[0]
print(first_congestion_object.level) # This might print "High"
```

Understanding these fields allows your application to make informed decisions, such as adjusting video quality, rerouting traffic, or notifying users about potential performance issues, based on real-time network conditions.

Question: What is the difference between network APIs and regular APIs?

Response:

Network APIs, particularly in the context of Nokia Network-as-Code (NaC), differ significantly from what are commonly referred to as "regular APIs" by exposing and enabling direct control over the underlying network infrastructure.

Here's a breakdown of the differences:

**Regular APIs (General-Purpose APIs):**
*   **Focus:** Primarily expose the functionalities, data, or business logic of a specific application, service, or database.
*   **Layer of Operation:** Operate at the application layer, facilitating communication between different software components.
*   **Control:** Allow developers to interact with and manipulate data or features *within* an application or service. They do not typically provide mechanisms to directly influence the network's behavior or resources.
*   **Examples:** APIs for payment gateways, social media platforms, mapping services, weather data, or CRM systems.

**Network APIs (as offered by Nokia NaC):**
*   **Focus:** Expose the native capabilities and resources of the network itself, making them programmable. This includes functionalities like Quality of Service (QoS), network slicing, device location, device status, and congestion insights.
*   **Layer of Operation:** Bridge the gap between applications and the network infrastructure, allowing applications to directly request and manage network resources.
*   **Control:** Enable developers to *program* and *control* the network's behavior. This means an application can dynamically request a guaranteed bandwidth (QoS on Demand), create a dedicated network slice for specific traffic, or retrieve real-time device location and network conditions.
*   **Abstraction:** Abstract the complexity of telecom protocols and infrastructure, making advanced network capabilities accessible to developers without specialized telecom expertise.
*   **Network Awareness:** Allow applications to become "network-aware," meaning they can understand and react to real-time network conditions and limitations.
*   **Examples:** Nokia NaC's Quality of Service on Demand API, Network Slicing API, Device Location API, Device Status API, and Congestion Insights API.

In essence, while regular APIs allow applications to interact with other applications or data sources, Network APIs empower applications to interact with and influence the network itself, transforming it from a static "black box" into a dynamic, programmable resource. This enables applications to optimize their performance, enhance user experience, and unlock new use cases by tailoring network behavior to their specific needs.

Question: What do I need to do to my application in order to be able to receive notifications from Network as Code?

Response:

To receive notifications from Nokia Network-as-Code (NaC), your application needs to be configured to act as a webhook receiver. This involves setting up a publicly accessible HTTP endpoint that NaC can send `POST` requests to when an event occurs.

Here's what you need to do:

1.  **Set up an HTTP Endpoint (Webhook):**
    Your application must expose an HTTP endpoint (a URL path) that can receive incoming `POST` requests. This endpoint will serve as the `notification_url` that you provide to NaC when subscribing to events. The exact implementation of this endpoint can be handled by developers as they see fit, using frameworks like FastAPI for Python or Express for TypeScript.

2.  **Configure `notification_url` and `notification_auth_token`:**
    When you subscribe to a specific NaC event (e.g., device connectivity status, roaming status, QoD session updates, slice lifecycle events, congestion level notifications), you will specify this `notification_url` and a `notification_auth_token`. The `notification_auth_token` is a shared secret that NaC will include in the `Authorization` header of the `POST` request it sends to your endpoint.

    *Example of subscribing to device connectivity notifications in Python:*
    ```python
    import network_as_code as nac
    from network_as_code.models.device import Device, DeviceIpv4Addr

    client = nac.NetworkAsCodeClient(token="<your-application-key-here>")

    my_device = client.devices.get(
        "device@testcsp.net",
        ipv4_address=DeviceIpv4Addr(
            public_address="233.252.0.2",
            private_address="192.0.2.25",
            public_port=80
        ),
        ipv6_address="2001:db8:1234:5678:9abc:def0:fedc:ba98",
        phone_number="+36721601234567"
    )

    my_subscription = client.connectivity.subscribe(
        event_type="org.camaraproject.device-status.v0.connectivity-data",
        device=my_device,
        max_num_of_reports=5,
        notification_url="https://example.com/notify", # Your application's endpoint
        notification_auth_token="replace-with-your-auth-token" # Your shared secret
    )
    ```

3.  **Implement a Notification Handler:**
    Your application's endpoint must contain logic to process the incoming `POST` request. The body of this request will be a JSON object representing the notification. This object typically includes fields like `id`, `source`, `type`, `time`, and a `data` field containing the specific event details.

    *Example of a Python handler using FastAPI:*
    ```python
    from fastapi import FastAPI, Header
    from pydantic import BaseModel
    from typing import Union
    from typing_extensions import Annotated

    app = FastAPI()

    class Data(BaseModel):
        # Define the structure of the 'data' field based on the event type
        # For example, for device status, it might contain device info and subscriptionId
        device: dict # Simplified for example
        subscriptionId: str

    class Notification(BaseModel):
        id: str
        source: str
        type: str
        specversion: str
        datacontenttype: str
        time: str
        data: Data

    @app.post("/notifications")
    def receive_notification(
        notification: Notification,
        authorization: Annotated[Union[str, None], Header]
    ):
        if authorization == "Bearer my-token": # Validate the token
            # Process the notification object
            print(notification)
            # React to the notification based on its content
            return {"status": "success"}
        else:
            return {"status": "unauthorized"}, 401
    ```

4.  **Authentication:**
    It is crucial to validate the `Authorization` header of the incoming `POST` request. NaC will send the `notification_auth_token` you provided during subscription in this header (e.g., `Bearer my-token`). Your handler should check if this token matches your expected secret to ensure the notification's authenticity.

Question: What are network APIs and why should I use them?

Response:

Network APIs, within the context of Nokia Network-as-Code (NaC), are interfaces that expose the capabilities of a telecommunications network, making them programmable and accessible to application developers. Traditionally, networks have been "black boxes" and static, with applications having limited awareness or control over network conditions. Network APIs change this by simplifying network programming and allowing developers, even those without deep telecom experience, to interact with native 5G capabilities in an easier and more efficient way.

You should use Network APIs for several compelling reasons:

1.  **Simplified Network Programming:** They abstract the complexity of the underlying network infrastructure, allowing developers to integrate advanced network functionalities into their applications using familiar API calls and SDKs, rather than requiring specialized telecom knowledge.
2.  **Network Awareness for Applications:** Applications can become aware of real-time network conditions, such as Quality of Service (QoS), congestion, or device status. This enables applications to dynamically adapt and optimize their behavior based on the network environment.
3.  **Dynamic Resource Control:** Network APIs provide the ability to request and manage network capabilities on demand. This includes features like Quality-of-Service on Demand (QoD) to prioritize specific data flows, control uplink and downlink ratios, or provision specialized network slices for critical applications.
4.  **Cost Optimization:** By allowing applications to dynamically request premium network capabilities only when needed, Network APIs enable a more efficient and cost-effective use of network resources. You pay for the specific capabilities you consume, rather than relying on static, high-cost service plans.
5.  **Improved User Experience:** The ability to dynamically manage QoS, ensure guaranteed latency, and adapt to changing network conditions leads to a more consistent, reliable, and high-quality experience for end-users, especially for demanding applications like gaming, VR, and live streaming.
6.  **Enhanced Security and Fraud Prevention:** APIs like SIM Swap, Number Verification, and Location Verification offer robust tools to improve security, authenticate users, and prevent fraudulent activities in various applications, particularly in financial services.
7.  **Innovation and New Application Development:** Network APIs unlock the potential for creating entirely new categories of innovative, network-aware applications across diverse industries. This includes solutions for smart mobility, industrial automation, remote healthcare, immersive entertainment, and more, by leveraging the full capabilities of programmable networks.
8.  **Operational Efficiency:** For enterprises, Network APIs can streamline operations, improve worker safety, and enhance asset management by providing real-time device status, location insights, and dynamic network adjustments.

In essence, Network APIs transform networks from static infrastructure into flexible, programmable platforms that applications can actively interact with, leading to more intelligent, efficient, and innovative digital services.

Question: What is the difference between network APIs and regular APIs?

Response:

The distinction between "network APIs" and "regular APIs" lies primarily in the domain they expose and the underlying infrastructure they interact with.

**Regular APIs (Application Programming Interfaces)** are a broad category of interfaces that allow different software applications to communicate and exchange data. They typically expose functionalities of a software service, a database, or an application. For example:
*   A payment gateway API allows an e-commerce site to process transactions.
*   A social media API allows an app to post updates or retrieve user data.
*   A mapping API provides location data or routing services.
These APIs primarily deal with software-level interactions, data management, and application logic.

**Network APIs**, specifically in the context of Nokia Network-as-Code (NaC), are a specialized type of API designed to expose and make programmable the capabilities of the underlying telecommunications network infrastructure. Instead of interacting with a software application or database, Network APIs allow developers to directly interact with and influence the behavior of the 5G network, its core functions, and its resources.

Here are the key differences:

1.  **Domain of Exposure:**
    *   **Regular APIs:** Expose functionalities of software applications, databases, or web services.
    *   **Network APIs:** Expose functionalities of the network itself, such as Quality of Service (QoS), device location, network slicing, device status, and security features like SIM Swap detection.

2.  **Underlying Infrastructure:**
    *   **Regular APIs:** Interact with application servers, databases, cloud services, etc.
    *   **Network APIs:** Interact with core network elements (like the 5G Core), radio access networks, edge clouds, and other telecom infrastructure, often facilitated by components like the Network Exposure Function (NEF).

3.  **Capabilities and Control:**
    *   **Regular APIs:** Enable data exchange, trigger application-specific actions, or retrieve information from a software system.
    *   **Network APIs:** Allow applications to actively request, control, and adapt network resources and behavior. For example, an application can request a temporary bandwidth boost (QoS on Demand), create a dedicated network slice for critical traffic, or verify a device's location directly from the network.

4.  **Complexity Abstraction:**
    *   Both types of APIs abstract complexity. However, **Network APIs** specifically abstract the immense complexity of telecom protocols, standards, and infrastructure, making advanced 5G capabilities accessible to developers who may not have deep telecommunications expertise.

5.  **Impact on Application Behavior:**
    *   **Regular APIs:** Enhance an application's features by integrating external services or data.
    *   **Network APIs:** Enable applications to become "network-aware," allowing them to dynamically react to and optimize their performance based on real-time network conditions, leading to better user experience and resource utilization.

In summary, while both are interfaces for programmatic interaction, regular APIs focus on software-to-software communication, whereas Network APIs specifically bridge the gap between applications and the underlying network infrastructure, making the network itself a programmable resource.
