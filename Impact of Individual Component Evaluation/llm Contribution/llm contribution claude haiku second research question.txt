Question: Why do I need to include private, public IP addresses and port when creating a QoD session?

The key points are:

1. Network Address Translation (NAT) Considerations:
   - Devices connected to the internet often have a public IP address that is different from their private, internal IP address.
   - This is due to the use of Network Address Translation (NAT) technology, which is commonly employed by network routers and gateways.

2. Identifying Devices:
   - The NaC platform needs to be able to accurately identify the device for which the QoD session is being created.
   - Relying solely on the public IPv4 address is not sufficient, as multiple devices behind the same NAT router may share the same public IP address.

3. Providing Additional Identifiers:
   - To uniquely identify a device, the NaC platform requires the device's public IP address, private IP address, and public port.
   - This combination of identifiers ensures that the device can be accurately recognized, even in the presence of NAT.

4. Improved Accuracy and Reliability:
   - By using the public and private IP addresses, along with the public port, the NaC platform can more reliably and accurately identify the target device for the QoD session.
   - This helps to ensure that the QoD session is properly established and associated with the correct device.

In summary, the need to provide both public and private IP addresses, as well as the public port, when creating a QoD session is a result of the way devices are identified in the NaC platform, taking into account the common use of NAT technology in modern network environments.

4. Device and Identity Management:
   - NaC also provides mechanisms for managing device identities, such as phone numbers or email-like identifiers.
   - This allows the end-user to authorize specific devices and control which applications can access their device data.

5. Consent Validation:
   - When an application makes an API call, NaC validates that the call is authorized by the end-user's consent, ensuring that the application is only performing actions within the granted scope.

By following this consent management process, the NaC platform ensures that end-user privacy is protected and that applications can only access device data and perform actions with the explicit consent of the device owner.

Question: What is a code example that uses QoD API?

Response:

Here's an example of how you can use the Quality of Service on Demand (QoD) API in Network as Code (NaC) using Python:

```python
import network_as_code as nac

# Create a Network as Code client
client = nac.NetworkAsCodeClient(token="<your-application-key-here>")

# Create a device object
device = client.devices.get(
    ipv4_address=nac.models.DeviceIpv4Addr(
        public_address="233.252.0.2",
        private_address="192.0.2.25",
        public_port=80
    ),
    ipv6_address="2001:db8:1234:5678:9abc:def0:fedc:ba98",
    phone_number="+36721601234567"
)

# Create a QoD session
qod_session = device.create_qod_session(
    service_ipv4="233.252.0.2",
    service_ipv6="2001:db8:1234:5678:9abc:def0:fedc:ba98",
    profile="QOS_L",
    duration=3600,
    notification_url="https://notify.me/here",
    notification_auth_token="replace-with-your-auth-token"
)

# Print the QoD session details
```

Question: What is the difference between network APIs and regular APIs?

Response:

The main difference between network APIs and regular APIs lies in the scope and focus of the functionality they provide:

1. Scope and Functionality:
   - Regular APIs typically provide access to application-specific functionality, data, or services. They are focused on the business logic and features of a particular application or platform.
   - Network APIs, on the other hand, provide access to network-specific capabilities, resources, and telemetry. They allow developers to programmatically interact with and control various aspects of the underlying network infrastructure.

2. Network Awareness:
   - Regular APIs are generally agnostic to the network conditions and do not directly consider the network's performance, availability, or other characteristics.
   - Network APIs are designed to be network-aware, meaning they can provide insights into the network state, allow dynamic adjustments based on network conditions, and enable network-centric use cases.

3. Use Cases:
   - Regular APIs are commonly used to build business applications, integrate systems, and enable data exchange between different components or services.
   - Network APIs are typically used to build network-aware applications, optimize network resource utilization, enable new network-based services, and provide visibility and control over the network infrastructure.

4. Abstraction Level:
   - Regular APIs often provide a higher-level abstraction, hiding the underlying technical complexities from the developers.
   - Network APIs may provide a lower-level of abstraction, exposing more granular network-specific details and capabilities, allowing developers to have more control and customization over the network behavior.

5. Ecosystem and Stakeholders:
   - Regular APIs are usually developed and consumed by application developers, business users, and IT teams within an organization.
   - Network APIs are often developed and consumed by a broader ecosystem, including network operators, service providers, application developers, and even end-users, depending on the use case.

In summary, network APIs are specifically designed to provide programmatic access to network-centric capabilities, resources, and telemetry, enabling developers to build network-aware applications and leverage the network as a platform for innovation. Question: What is the difference between network APIs and regular APIs?

Response:

The main difference between network APIs and regular APIs lies in the scope and focus of the functionality they provide:

1. Scope and Focus:
   - **Regular APIs**: These are typically application-centric, focusing on exposing the functionality and data of a specific application or service. They allow applications to interact with each other and share information.
   - **Network APIs**: These are network-centric, focusing on exposing the capabilities and resources of the underlying network infrastructure. They allow applications to directly interact with and leverage the network's capabilities.

2. Network Awareness:
   - **Regular APIs**: These APIs are generally unaware of the network conditions and do not provide any network-specific functionality.
   - **Network APIs**: These APIs are designed to be network-aware, providing information about the network state, performance, and resources. They also allow applications to control and manage network-level parameters.

3. Use Cases:
   - **Regular APIs**: These are commonly used for integrating different applications, building composite services, and enabling data exchange between systems.
   - **Network APIs**: These are used to create network-aware applications, optimize application performance, enable new network-centric services, and improve network resource utilization.

4. Examples:
   - **Regular APIs**: APIs for e-commerce platforms, social media platforms, content management systems, etc.
   - **Network APIs**: APIs for quality of service (QoS) management, device location tracking, network slicing, network insights, and more.

5. Abstraction Level:
   - **Regular APIs**: These APIs typically provide a higher-level abstraction, hiding the underlying complexities of the system.
   - **Network APIs**: These APIs provide a lower-level abstraction, exposing more granular network-level details and controls.

In summary, network APIs are specifically designed to enable applications to directly interact with and leverage the capabilities of the underlying network infrastructure, whereas regular APIs focus on application-level functionality and data exchange. By using network APIs, developers can create more network-aware, performant, and innovative applications that can better utilize and optimize network resources.
